# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Setup

```bash
bash build_nvim.sh   # optional: build latest Neovim from source (>= 0.11 required)
bash install.sh      # installs tools and symlinks nvim/ → ~/.config/nvim
```

Plugins are auto-installed by Lazy.nvim on first start. The curated Treesitter parser set (bash, c, cpp, lua, python, rust, vim, vimdoc, json, toml, yaml, markdown, markdown_inline) is then installed automatically too, via the `LazyDone` autocmd in `lua/config/treesitter.lua` — no manual step needed. To add a parser for another language, run `:TSInstall <language>` (e.g. `:TSInstall go`); the bare `:TSInstall!` with no arguments errors with `E471: Argument required`, since this plugin's command requires explicit parser names.

## Architecture

The config lives entirely in `nvim/` (installed as `~/.config/nvim`). Load order in `init.lua`:

1. `lua/globals.lua` — global vars, leader key (`,`), disabled built-in plugins
2. `viml_conf/options.vim` — Vim options (tabs, folds, UI)
3. `lua/custom-autocmd.lua` — autocommands (yank highlight, auto-reload, large-file handling, etc.)
4. `lua/custom-commands.lua` — user commands: `:Format`, `:LspInfo2`, `:ReloadConfig`, `:LspRestart`, `:MesonSetup`, `:MesonBuild`, `:MesonLink`, `:GccDebug`, `:KernelSetup`, `:OEPkgSetup`
5. `lua/mappings.lua` — all keymaps not owned by a plugin
6. `viml_conf/plugins.vim` → `lua/plugin_specs.lua` — Lazy.nvim plugin list
7. `lua/diagnostic-conf.lua` — LSP diagnostic display
8. `lua/colorschemes.lua` — colorscheme loader

**Plugin configs** live one-file-per-plugin in `nvim/lua/config/`. Each is loaded from the plugin's `config = function() require("config.<name>") end` in `plugin_specs.lua`.

**Snippets** are handled entirely by LuaSnip: custom snippets live in `nvim/lua/snippets/*.lua` (loaded via LuaSnip's `from_lua` loader), and VSCode-format snippets come from `friendly-snippets`.

**VimScript utilities** in `nvim/autoload/` (`utils.vim`, `buf_utils.vim`, `text_obj.vim`) are called from Lua via `vim.fn["utils#..."]()`.

## Key Conventions

- **Leader key**: `,`
- **`c` / `C` / `cc`** in normal mode are remapped to `"_c` (black-hole register) — changes do not overwrite the yank register.
- **`<C-h>` in insert mode** is `<BS>` (not window navigation — that's normal mode only).
- **`<C-j>` / `<C-k>` in insert mode** are reserved for LuaSnip expand/jump; do not bind them to anything else.
- **`<leader>c` prefix** is reserved for cscope maps — do not add other mappings under `<leader>c`.
- **oil.nvim** has `<C-h>` keymap intentionally omitted to avoid conflicting with `<C-w>h` window navigation.

## Changing the Colorscheme

Edit the last line of `init.lua`:
```lua
color_scheme.select_colorscheme("everforest")  -- keys defined in lua/colorschemes.lua
```
Available keys: `everforest`, `gruvbox_material`, `sonokai`, `tokyonight`, `catppuccin`, `kanagawa`, `nightfox`.

## Adding a Plugin

Add a spec entry to the appropriate section in `lua/plugin_specs.lua`, then create `lua/config/<plugin-name>.lua` for its config if non-trivial. Lazy-load with `event`, `cmd`, `ft`, or `keys` to keep startup fast.

## Meson / C++ Workflow

`:MesonSetup [pkgdir ...]` — runs `meson setup builddir` and symlinks `compile_commands.json` to the package root, then restarts LSP. Supports globs (e.g., `:MesonSetup ./packages/*`). Keymaps: `<leader>ms` (setup), `<leader>mb` (build), `<leader>ml` (re-link only).

For kernel-style C formatting, place a `.clang-format` at the project root with `BasedOnStyle: Linux`.

## LSP Notes

`:LspRestart` is a custom reimplementation — the built-in was removed in Neovim 0.11. It stops all clients and re-enables them per buffer.

clangd requires `compile_commands.json` at the project root (generated by `:MesonSetup` or `cmake -DCMAKE_EXPORT_COMPILE_COMMANDS=ON`).
